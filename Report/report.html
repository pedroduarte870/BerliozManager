No âmbito do perfil Processamento de Linguagens e Conhecimento, do Mestrado em Engenharia Informática, este relatório pretende apresentar o Projecto de um Gestor de Audições/Concertos Musicais, adotando os conhecimentos adquiridos nas unidades curriculares de Gramáticas na Compreensão de Software e Processamento e Representação de Informação, bem como as decisões que foram tomadas e as suas dificuldades.

Este trabalho tem como objetivo principal desenvolver uma solução em base de dados para preencher uma necessidade específica de uma escola de música que ao longo de um ano letivo poderá ter que organizar dezenas ou centenas de audições e concertos.

Estrutura do Relatório

Numa primeira fase, é apresentado o Resumo deste trabalho prático onde é descrito de forma bastante sucinta o problema e a motivação para encontrar a sua solução.

Introdução

Contextualização

Vivemos tempos difíceis com restrições a todos os níveis e com o tempo super acelerado. Neste contexto, ser-se competitivo é cada vez difícil e exigente.

Quando se pensa no mercado de desenvolvimento de software é cada vez mais difícil descobrir nichos de mercado ainda virgens. No entanto, é sempre possível descobrir nichos de necessidades específicas: muitas vezes as instituições adquirem grandes soluções que estão concebidas para o grande mercado genérico e que descuram as necessidades mais específicas. Ora, se rapidamente fosse possível desenvolver um componente que interoperasse com a grande aplicação e que resolvesse aquelas necessidades específicas poderiam sair daqui alguns pequenos produtos com uma economia de escala associada.

É neste contexto microaplicacional que se propôs este projeto.

Para tal fixou-se um tema que resultou de algumas conversas e vivências de um dos docentes: uma escola de música. Já existem várias aplicações para a gestão de escolas mas nenhuma resolve bem as especificidades de uma escola de música. Assim, depois de uma reunião com a direção de uma escola de música identificaram-se algumas necessidades específicas que poderão ser solucionadas por microaplicações: um gestor de audições/concertos e um gestor de programas específicos.

Caso de Estudo

O trabalho prático consiste no desenvolvimento de um gestor de dados, sendo populado com a informação contida em ficheiros XML, através de formulários e também através de uma notação rápida para criar audições escrita numa linguagem desenvolvida para esse efeito.

A concretização deste projeto pressupõe a realização de um conjunto etapas. A realçar:

  • Analisar os dados fornecidos pela equipa docente e os objetivos a atingir. Perceber as questões fundamentais do domínio do problema assim como tomar decisões quanto ao desenho da base dados e principais requisitos a cumprir.

  • Decidir que metodologias e ferramentas usar.

  • Desenhar o modelo da base dados, de maneira a poder armazenar e relacionar toda a informação importante presente nos arquivos de forma a satisfazer os requisitos do problema.

  • Analisar e trabalhar cada fonte de dados separadamente. Processando e inserindo a informação na base de dados.

  • Construção de uma interface gráfica para mostrar informação de forma simples.

Objetivos

Como foi mencionado anteriormente, o foco deste projeto passa por desenvolver um sistema capaz de gerir informação relativamente a uma escola de música, no que toca às suas inumeras audições e concertos que realizam ao longo do ano lectivo. E através dos conhecimentos adquiridos nas aulas, processar e validar essa informação e inserir informaticamente na base de dados. Tendo em conta os requisitos deste projeto, podemos dizer que o principal objetivo é criar, alterar, listar e consultar uma audição a partir de informação fornecida pelo utilizador;

A aplicação deverá garantir a persistência e consistência de dados com auxílio de um sistema de base de dados relacional.

Ferramentas

Para o desenvolvimento do nosso sistema precisamos de combinar um vasto conjunto de tecnologias que nos permitam simplificar o nosso trabalho.

Para gerir e desenvolver paralelamente o código utilizámos o Github, que é um serviço de web hosting para o sistema de controlo de versões Git. Com a ajuda desta tecnologia, é mais fácil para o grupo discutir o código produzido, melhora-lo e juntar ao código já existente.

Em relação ao desenvolvimento, recorremos ao phpMyAdmin que é um aplicativo web desenvolvido em PHP para administração do MySQL pela Internet. A partir deste sistema é possível criar e remover bases de dados, criar, remover e alterar tabelas, inserir, remover e editar campos, executar códigos SQL e manipular campos chaves.

De forma facilitar o desenvolvimento da interface gráfica da aplicação web, utilizamos o Bootstrap que é uma coleção livre e open-source de ferramentas para criar sites e aplicações web. Ele contém HTML e modelos de design baseados em CSS para tipografia, formulários, botões, navegação e outros componentes, bem como extensões JavaScript opcionais. .

No que diz respeito a manipulação de documentos, e também para a escrita deste relatório utilizámos as tecnologias dadas nas aulas tais como DTD, XSLT e XSD.

Usámos ANTLR, uma gramática de atributos e a liguagem de programação genérica Java

Arquitectura usada


Arquitectura BerliozManager

Existe uma base de dados sql cujo modelo relacional foi criado usando o programa Visual Paradigm que injectou uma estrutura de dados, em baixo explicada, com base nos datasets fornecidos pelo docente.

Com esta base de dados comunicam os controladores do web site defenidos na linguagem PHP e a aplicação Java de suporte ao planeamento de audições com auxilio do parse generator ANTLR.

A comunicação que existe entre a aplicação java e a base de dados SQL destina-se à inserção da informação que diz respeito a audições bem definidas. Outro papel da aplicação java é gerar ficheiros XML e TXT também com informação de audição para mais tarde o controlador do site usar para gerar ficheiros HTML e PDF

A parte que o utlizador visualiza, denominada front-end, foi desenvolvida em HTML com auxilio a bibliotecas de templates css Bootstrap. A correspondência de dados que se disponibiliza é obtida através da comunicação com o back-end com recurso a AJAX realizando POST's dos dados à medida que o utlizador realiza uma acção. O controlador em PHP analisa o request do front-end e devolve na maioria das vezes os dados pedidos em JSON que mais tarde são processados pelo front-end

Para os controladores comunicarem com a base de dados em SQL usa-se o mysqli para realizar querys podendo assim inserir, actualizar, apagar e consultar dados.

Identificação dos relacionamentos entre entidades

Depois de identificadas as principais entidades do sistema é necessário perceber quais as relações existentes entre cada entidade.

Curso(1) - tem - Aluno(N)

Este relacionamento traduz o facto de um Curso ter vários alunos, mas um aluno estar inscrito a um e apenas um curso

Grupo(N) - tem - Alunos(M)

Um grupo é constituido por vários alunos. Mas um aluno pode estar em vários grupos

Curso(1) - tem - Instrumento(1)

A cada curso é atribuído um instrumento

Curso(1) - tem - Professor(N)

Um curso é leccionado por vários professores

Audição(1) - tem - ActuaçõesN)

Uma audição é constituída por várias actuações

Actuação(1) - tem - Peça(N)

Uma actuação é constituída por várias peças

Compositor(1) - tem - Peça(N)

Um compositor é autor de várias peças

Actuação(1) - pode ter - Alunos(1)

Uma actuação pode ser realizada por um aluno.

Actuação(1) - pode ter - Grupo(1)

Uma actuação pode ser realizada por um grupo de alunos.

Desenvolvimento e Funcionalidades

O nome escolhido para a nossa aplicação foi Berlioz Managercomo forma de homenagem a Hector Berlioz, um músico que teve contribuições significativas para a orquestra moderna. Ele criou música para enormes grupos orquestrais, e realizou vários concertos com mais de mil músicos. Também compôs cerca de cinquenta canções

Decidimos dividir a aplicação em duas distintas páginas: Uma pensada para a visualização do público em geral, onde é possível consultar os próximos eventos da escola (audições/concertos), uma breve descrição do trabalho realizado, como o contacto do grupo que realizou o trabalho.

Notas Finais

Conclusão

Em conclusão, vemos este projeto como mais um forma de pôr em prática os conhecimentos adquiridos e as ferramentas disponibilizadas no perfil de Processamento de Linguagens e Conhecimento durante o 1º Semestre. Como o enunciado deste trabalho prático implica um domínio de todos os conceitos abordados nas aulas, este ajudou a consolidar os conhecimentos e a combater algumas dificuldades que tínhamos anteriormente.

Quanto à fase de desenvolvimento do projecto, podemos considerar que a fase de interpretação do enunciado tomou algum tempo ao grupo de forma a chegar a um acordo quando à implementação da base de dados, nesta fase da criação da base de dados deparamo-nos com algumas dificildades iniciais, mas apartir do momento em que se conseguiu povoar a base de dados, tornou mais facéis as tarefas seguintes do grupo. Na parte final do desenvolvimento, com uma menor disponibilidade de tempo dos elementos do grupo, alguns dos objectivos iniciais não foram cumpridos. Pelo que na segunda oportunidade de apresentação os requesitos estão quase na totalidade implementados

Depois de uma avaliação ponderada, o grupo considerou os requisitos pedidos foram cumpridos, talvez numa abordagem inicial com o auxilio de uma framework ajudava o grupo em termos de tempo e preocupação com os métodos CRUD. Mas fica a certeza de seria um bom projecto para continuar a trabalhar de forma a terminar-lo e o disponibilizar ao sujeito cliente, neste caso uma escola de música.
